### Pipeline Stages (9 Tasks in Sequence)

```
data/landing
    ↓
[01_ingest_files] → data/raw/{run_id}/manifest.yaml
    ↓
[02_validate_raw_schema] (Pandera)
    ↓
[03_profile_data] → reports/profile_*.html
    ↓
[04_clean_data] → data/interim/{run_id}/
    ↓
[05_engineer_features] → data/features/{run_id}/{train,test}.parquet
    ↓
[06_validate_features_schema] (Pandera)
    ↓
[07_train_models] → MLflow runs (Ridge, LightGBM)
    ↓
[08_register_to_mlflow] → Model Registry (Staging stage)
    ↓
[09_drift_report] → reports/drift_*.html
```

### Model Lifecycle

```
Training
  ↓
MLflow Experiment Run
  ↓
Register to Model Registry (Staging)
  ↓
Manual Promotion to Production (via MLflow UI)
  ↓
FastAPI Server Loads Production Model on Startup
  ↓
/predict endpoint serves predictions
```

## Key Design Decisions

1. **Single DAG, 9 Sequential Tasks**: One pipeline with clear stage separation (ingest → validate → profile → clean → engineer → validate → train → register → drift)

2. **Manifest-Based Versioning**: YAML manifests at each storage boundary (raw, interim, features) contain file hashes, row counts, and timestamps for data lineage

3. **Pandera at Boundaries**: Schema validation enforced at:
   - Raw data ingestion
   - Cleaned interim data
   - Feature matrix creation
   - Prevents invalid data from flowing downstream

4. **No Auto-Promotion to Production**: All models register to MLflow Staging. Manual UI click required for Production promotion. Prevents accidental model rollouts.

5. **LocalExecutor for POC**: Single-machine Airflow executor (no Celery, no Kubernetes yet). Suitable for POC validation; scales to KubernetesPodOperator in future.

6. **Separate PostgreSQL Instances**: 
   - Airflow metadata DB (port 5432)
   - MLflow tracking DB (port 5433)
   - Prevents schema conflicts

7. **FastAPI with Read-Only Artifacts**: FastAPI mounts MLflow artifacts as read-only volume. Prevents accidental writes; enables horizontal scaling.

8. **On-Demand Drift Reporting**: Drift monitoring runs as final DAG task (not continuous). Compares current features to previous training set if available.

## How to Use This POC

### 1. Setup (5 minutes)
```bash
cp .env.example .env
docker-compose up -d
```

### 2. Verify Services (1 minute)
```bash
docker-compose ps  # All services should be healthy
curl http://localhost:8080  # Airflow webserver
```

### 3. Test Pipeline (10-30 minutes)
Follow [TESTING.md](TESTING.md):
- Place CSVs in `data/landing/`
- Trigger DAG via Airflow UI
- Monitor execution
- Check outputs at each stage

### 4. Serve Predictions (5 minutes)
- Promote a model to Production in MLflow
- Test `/health` and `/predict` endpoints
- Validate predictions make sense

### 5. Extend / Customize
- Modify `config/` YAML files to adjust recipes
- Update schemas in `src/schemas/` as data evolves
- Add new feature engineering steps in `src/features.py`
- Scale to Kubernetes using KubernetesPodOperator
